Why Do Good Programmers Write Bad Code - Part 1

09 Nov 2017 by Ozan Gulle

Introduction

“Why do good programmers write bad code” is a question I’ve been pondering on for quite a while now. The question doesn’t come out of nowhere. Every now and then when I’m reading code at work, I stumble upon pieces of abysmal code written by people whom I know to be great programmers. I also stumble upon abysmal code written by myself every now and then, which is a very discomforting experience. I believe almost every programmer would admit that they have written bad code at some point in their career. We have all written bad code at some point, sometimes even knowingly. When I say “good programmer” here, I mean someone who under normal circumstances writes good code, someone capable of solving the problems that you throw at them. What I’m interested in finding out is: Why does someone who usually writes good code suddenly start writing bad code.

This post turned out to be quite research intensive and long, so I ended up splitting it into two posts. In this one, I will discuss what bad code is and examine the reasons for it. The next post will be about strategies on avoiding these issues.

Bad Code

What exactly is bad code? Well, it’s basically code that isn’t good. It sounds like a cheap redundant answer, but I want to argue that it is not. I think it is much easier to define bad code as opposed to good code and then define the latter explicitly. This is because there are usually a handful ways of getting something right, but a lot of things can go “bad” in the process. Sure, there are usually multiple ways of doing things right in a given scenario, but it usually boils down to a handful of methods. These methods endured the test of time and were proven to be successful if they are employed correctly. This proven methodology is where the “engineering” part in many programmer titles comes from. The use of this word is definitely not without opposition, but it serves a right purpose.

I will not try to define good or bad code here since not only would that take too much space, but it is also a topic examined by numerous people before me. Without going into definitions, I want to refer to a great talk from Raquel Velez (npm) about bad code. In her speech during JavaScript NZ, 2017, she described and categorized bad code as follows:

  1. functionality
    1. malicious/immoral code
    2. non-functional/partially-functional/buggy code
    3. code with security flaws
    4. code that doesn’t handle edge cases
  2. maintainability
    1. code with bad architecture aka. “spaghetti” code
    2. code with unknown side effects aka. false advertising in code
  3. readability
    1. cryptic/“clever” code
    2. really long functions
    3. code that doesn’t match styling standards

For the sake of not further expanding the scope of this post, I’m leaving out the first and the last elements on this list – malicious code and code that doesn’t match styling standards – since the former is a completely different topic on its own and the latter is a styling issue as the name suggests. What am interested in is a) non-functional or partially functional code, b) unmaintainable code, c) unreadable code, or d) any combination of these.

All of these are detrimental because they affect the quality of a project with a severity level that corresponds to their position on the list. If there is something functionally wrong with the code, it would not work as intended. This is also the only category of bad code where the users themselves may realize that the code must have been poorly written. This is why it is the worst kind of bad code you can have. If your codebase is just unmaintainable, it is at least working as intended. As long as you don’t need to extend the functionality or encounter any bugs, you can leave it running. It is, of course, a huge liability because it is very difficult to debug and no one in your team would voluntarily go anywhere near it. The least severe kind of bad code is the one that is difficult to read. This kind of code ends up making your project slower to debug or extend, thus making it more expensive.

Reasons for Writing Bad Code

It turns out that categorizing bad code is much easier than finding out why bad code happens. The methodological difficulty arises from the fact that nobody pays multiple teams of programmers over several months or years to conduct carefully designed experiments. Every paper written about this topic is based on real world examples where there is no control group and there are numerous variables involved in the outcome.

I also want to point out that I am not interested here in finding out why IT projects fail. You can have people writing bad code in a successful IT project, or a project completely fail without any bad code smells. There are very interesting articles out there such as this one here which investigate why certain projects fail. It is admittedly a very interesting topic, but it is not what’s being examined here.

In their research Why Good Developers Write Bad Code: An Observational Case Study of the Impacts of Organizational Factors on Software Quality Lavallée & Robillard (2015) observed a development process over two years in a well established company. They list ten causes which affect the quality. Lavallée & Robillard (2015) never explicitly state what they mean with “quality”, and I take that they use the word with regards to functionality. Since they observed a single team from a reputable company for this paper, I’m assuming that maintainability and readability were not much of a problem (but you never know). Nevertheless, I do think that the points they mention may also cause maintainability and readability problems in code. Since functionality has a higher priority in a project, if the severity of the problems have come so far, I would expect the best practices of maintainability and readability to have been already sacrificed.

The ten causes that affect the quality in their study are: 1) documenting complexity, 2) internal dependencies, 3) external dependencies, 4) hosting by third parties, 5) organically grown processes, 6) budget protection, 7) scope protection, 8) organizational politics, 9) human resource planning, and 10) undue pressure. I would like to leave out several of the points above, since these are issues that affect the overall quality of the project, but don’t necessarily produce bad code as I defined it. furthermore, I think it is possible to concentrate the arguments they provide into four main issues which can be then further grouped under two main categories:

  1. Organizational problems on the company/management level
    • Human resource planning: This refers to freelancers who leave the project prematurely because their contracts expire. With them goes the knowledge that they have acquired during the project. This point can be expanded to any kind of changes in the team structure during the project. Since the knowledge of the previous developers is gone, new developers might make the same mistakes without even realizing at that point that they are mistakes.
    • Undue pressure: Rushing the project, issuing threats, or setting unrealistic deadlines by the management. This is in my opinion the biggest contributor to bad code in this category. Especially readability but also often maintainability of the code fly straight out of the window the moment a project is rushed.
  2. Organizational problems in the development team
    • Documenting complexity: This refers partially to poorly documentation of the existing software or to the lack thereof completely, and partially to miscommunication between the business and the development teams, i.e. the documentation of the requirements. The lack of well-defined goals or poorly expressed goals may result in last minute patches and/or put the development team under pressure for missing deadlines
    • Organically grown processes: Undocumented processes that emerge organically without documentation. A classic example are the environment setup processes which are often undocumented, seldom communicated across different teams and may be subject to sudden changes depending on problems that arise during the development.

It is not surprising that the reasons for bad code are all related to organization and management. The question is “why do good programmers write bad code?”, so per definition, the competence of the individual programmers cannot be the root cause. None of the reasons for bad code are technical per se. So the answer to the question in the title is: Because of organizational and management reasons. How the team and the project is managed has a direct influence on not just the quality of the overall software as an end product, but also on the quality of the individual code snippets in this software.
