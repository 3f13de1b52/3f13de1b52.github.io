Why Do Good Programmers Write Bad Code - Part 2

14 Jun 2018 by Ozan Gulle

Summary of the First Part

In the first part of this article, I concluded that the main reason why good programmers write bad code is not due to technical issues but because of organizational problems on different levels. The emphasis here is on “good programmers”, meaning that lack of experience or skill is out of question. Bad code is defined roughly as (quoting from the first part)

  1. non-functional or partially functional code,
  2. unmaintainable code,
  3. unreadable code, or
  4. any combination of these
I concluded that since we are talking about competent and experienced programmers, the problems are not of technical nature, but occur due to organizational or management problems. The organizational issues which cause these different kinds of bad code can originate from company/management level or from the development team itself. Issues like human resource planning and undue pressure belong to the former category, whereas documentation complexity and organically grown processes belong to the latter one.

Solving the Problems

Human Resource Planning

This problem can be summarized with the bus (or truck) factor. The main question concerning the bus factor is “how many team members can get hit by a bus before the project is stalled”. For example, in a study done by Avelino G, Valente MT, Hora A. (2017) “What is the Truck Factor of popular GitHub applications? A first assessment.” (important: not peer-reviewed, methodology used may not be completely solid), the authors come to the conclusion that among the GitHub projects that they studied, they found out that 34% of the projects had a bus factor of 1 (meaning losing a single member would stall the project team) such as D3 and LESS.JS, and 30% of the projects had a bus factor of 2 such as CLOJURE, NETTY and CUCUMBER. In comparison, projects like LINUX and HOMEBREW had a bus factor of 130 and 250 respectively.

Lavallée & Robillard (2015) suggest knowledge sharing methods like pair programming and code reviews in order to spread the specialized knowledge in the team among its members. What they don’t seem to mention is the fact that there are various roles in a development team given a certain production cycle. Even if the knowledge required is spread among the team members evenly, the project can get stalled or heavily delayed simply because there aren’t enough members to fill all of the required roles. In any case, keeping the bus factor high in a team would result in a healthier development environment.

Undue Pressure

This is a very tricky problem to tackle. The reason why undue pressure happens is because someone from higher management – who is probably misinformed about the project – puts time related pressure on the development team. The speed of programming is more or less constant, with a somewhat increase overtime due to experience of the team. If someone puts an unrealistic deadline on the project, code quality has to suffer due to it.

There doesn’t seem to be a way around it other than informing the higher management about the risks about putting pressure on the team. As Lavallée & Robillard (2015) puts it, higher management is rarely interested in code quality. It might prove to be useful to educate them about technical debt and its risks on the long term. This, however, is not going to be successful against a higher management only thinking about short-term goals.

Documenting Complexity

According to Lavallée & Robillard (2015) documentation boils down to two main questions, namely what a certain piece of code does and why it exists. It is common knowledge that lack of documentation is one of the hallmarks of bad projects. With lack of documentation, I do not only refer to the lack of documentation as a separate entity on its own, but also to unreadable and cryptic code, or simply to code which is difficult to follow. If a documentation exists at all, another challenge arises: Keeping the documentation up-to-date, since code seldom remains the same for a long period of time.

This problem can be solved by a) encouraging a documentation culture throughout the project, b) enforcing code quality standards throughout the project, for example through code reviews, and c) generating documentation from code itself and removing the necessity to write separate documentation. There are many examples for this, like Compodoc for Angular projects.

Organically Grown Processes

This refers to processes which are completely specific to one project and don’t fit into the patterns used by the company. This might theoretically range from different naming conventions than the rest of the company to completely different build processes, dependencies, or architectural patterns. These processes do not only cause issues because they differ from rest of the company, but also because they are usually hastily prepared without proper documentation, vision, and/or plan. Often, they are simply makeshift solutions. The problem with organically grown processes can be solved with proper oversight over projects and a shift in development culture.
